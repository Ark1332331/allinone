# 学习痕迹记录

## 用途

这个文件记录用户在真实协作、学习、调试中的“原始理解资料”。

它不是稳定用户画像，也不是结论库。这里优先保存：

- 用户怎么描述问题
- 用户能准确感知到什么
- 用户暂时难以映射到什么代码/知识结构
- Codex 如何判断这个现象背后的能力缺口
- 下一次协作可以怎样降低反复试错
- 这件事对 allinone 产品设计有什么启示

## 记录原则

- 不把一次事件直接写成稳定画像。
- 只记录有证据的观察，不给用户贴泛化标签。
- 用户纠正后，以用户纠正为准。
- 只有多次重复出现、且用户认可的特质，才进入稳定画像或长期协作指南。
- 每条记录都要保留“是否进入稳定画像”的状态。

## 记录格式

```text
日期：
触发事件：
用户原话/现象：
观察到的能力或缺口：
证据：
下一次协作建议：
对产品设计的启示：
是否进入稳定画像：否 / 待验证 / 是
```

## 2026-05-18：PDF/LLM 回答锚点反复修错

日期：2026-05-18

触发事件：
PDF 原文件阅读页中，用户在右上角截图问答框、当前问答小框、LLM 回答文本中反复验证“选中文字 -> 提问 -> 原位置留下锚点 -> 点击锚点回到对应回答”的闭环。多次补丁修了表面现象，但右侧 PDF 截图回答里的选中文字追问仍没有留下可见锚点。

用户原话/现象：
用户持续指出“LLM 的回答里面选中文字提问追问对应的文字不会出现锚点”“点击锚点应该进入小框”“不要补丁了，我们需要解决问题”“能不能找出问题再改”。最后确认修复后，追问“为什么这里卡了这么久”，并进一步提出这对代码理解工具和自己的学习路径有启示。

观察到的能力或缺口：
用户能准确感知阅读体验的断裂，尤其是“阅读中提问不能打断阅读”“选区必须留下可回看的锚点”“回答文本也应该能继续成为提问对象”这些产品不变量。

当前缺口不主要是语法或单个 React API，而是“把 UI 现象映射到代码责任边界”的能力：哪些组件拥有哪段状态，哪个事件监听器处理了选区，为什么两个相似的回答框其实走了不同代码路径，为什么 z-index 还会受父级堆叠上下文限制。

证据：
本次真实根因是 `PdfReadingSurface` 的 PDF 局部回答选区路径与 `ReadingWorkspace` 的全局 document 选区路径竞争。用户看到的都是“回答里选中文字”，但代码里至少分成 PDF 截图回答框、当前问答小框、结构化 Markdown 正文、LLM 回答 Markdown 等不同路径。用户的自然语言需求没有错，但缺少“这是哪个容器/哪个事件路径”的中间层，导致 agent 前面多次修错位置。

下一次协作建议：
当同一交互问题连续失败两次后，Codex 必须暂停补丁，先写出：

1. 用户操作的具体场景
2. 可能涉及的组件和状态
3. 实际走到的事件路径
4. 预期应该走的事件路径
5. 准备修改的最小断点
6. 明确不修改的范围

用户可以用四件套快速约束问题：“场景 / 动作 / 预期 / 实际”。例如：“场景：PDF 原文件阅读页右上角第 1 页截图问答弹窗；动作：选中回答文本并追问；预期：追问回答出现，同时原回答文本留下锚点；实际：出现回答框但原回答无标记。”

对产品设计的启示：
allinone 的代码理解工具不应只是解释文件内容，而应能生成“交互链路图”：

```text
用户动作
-> 涉及组件
-> 事件监听器
-> 状态写入位置
-> 渲染位置
-> 可能断点
-> 验证方式
```

对用户当前阶段尤其有价值的学习方向是：

- 前端事件模型：捕获、冒泡、pointer/mouse 事件、stopPropagation
- React 状态和组件所有权：哪个组件拥有状态，为什么状态写错地方会没有锚点
- UI 堆叠上下文：z-index 为什么不总是全局比较
- 系统化调试：先定位断点，再改一处，并用测试或最小验证证明

是否进入稳定画像：待验证。

理由：
这次足以作为重要学习痕迹，但还不能直接写成稳定画像。后续如果用户在其他 UI/代码理解问题中也反复体现“产品体验判断强，但代码路径映射弱”，再提炼为稳定画像。
