# 多模型协作 Vibe Coding 工作流（通用版）

> Archived legacy workflow.
> Historical reference only. The active shared workflow now lives in `docs/workflows/`.

> **核心理念**: Opus 做架构师，Coding Model 做施工队，你做掌控者。
> **一次任务** = 一张 Task Spec → 一轮 TDD 实现 → 一次审查 → 一次合并。
>
> 适用：任何项目、任何语言、任何技术栈。复制此文件到项目 `docs/` 即可。

---

## 角色分工

```
Opus (架构师)             Coding Model (工程师)       你 (产品负责人)
──────────────────────────────────────────────────────────────────────
输入:  项目全局 + 你的需求    Task Spec                  自己的想法
产出:  Task Spec             代码 + 测试 + 完成报告      决策 / 验收
责任:  架构正确性            实现正确性                  需求正确性
边界:  不写实现代码          不改架构 / 不扩 Spec 范围   不写代码
```

---

## 单次任务循环（7 步）

```
你提需求
  ↓
Opus 理解项目 → 拆分任务 → 产出 Task Spec
  ↓
你审 Spec  ← 【掌控点 1：方向对不对】
  ↓  通过
Coding Model 拿 Spec → TDD 实现 → 自检 → 汇报
  ↓
Opus 代码审查（架构 + 风格 + 安全）
  ↓  通过
你验收  ← 【掌控点 2：实现对不对】
  ↓  通过
Opus 合并 + 更新文档
```

## 核心：Task Spec 是"合同"

Task Spec 是多模型协作的基石。Opus 写得越清楚，Coding Model 越不容易跑偏。

**一份好的 Spec 让 Coding Model 不需要理解整个项目就能开工。**

## 各阶段详解

### 1. 你提需求

告诉 Opus：
- 要做什么（用户视角）
- 为什么做（价值）
- 有什么约束

```
"用户反馈登录很慢。我想优化登录接口的响应时间，目标 1 秒以内。
不要动前端，只改后端。不要引入新的中间件。"
```

### 2. Opus 出 Spec

Opus 要做：
1. 读 CLAUDE.md → 了解技术栈和约束
2. 读相关代码 → 理解现有架构
3. 拆分任务 → 每任务 3-5 文件
4. 填 Task Spec 模板
5. 输出给你审

**判定标准**：Spec 足够详细吗？换一个 Coding Model 能直接开工吗？

### 3. 你审 Spec（掌控点 1）

审查清单：
- [ ] 涉及文件范围合理（不多不漏）
- [ ] 架构约束正确（不破坏现有规则）
- [ ] 禁止事项明确（边界清楚）
- [ ] 验收标准可量化（能明确判断"做完了"）
- [ ] 粒度合适（单任务 3-5 文件，不会太大也不会太小）

### 4. Coding Model 实现

严格按 Vibecoding 执行：

```
读 Spec → 读参考代码 → 写测试(RED) → 写实现(GREEN) → 重构 → 自检 → 汇报
```

**约束**：
- 只改 Spec 列出的文件
- 遵守 Spec 中的"禁止事项"
- 不"顺手优化" Spec 范围外的东西

**完成报告**（模板在 Spec 末尾），包含改动摘要、测试结果、自检清单。

### 5. Opus 代码审查

审查重点（不同于普通 CR）：

| 维度 | 检查什么 |
|------|---------|
| 架构合规 | 是否符合 Spec 中的架构约束 |
| 范围越界 | 是否修改了 Spec 外的文件 |
| 模式一致 | 风格是否与参考代码对齐 |
| 安全 | 硬编码密钥、未验证输入等 |
| 测试质量 | 覆盖了边界还是只测 happy path |

结论：**PASS**（进入验收）/ **REWORK**（退回修改）/ **BLOCK**（重设 Spec）

### 6. 你验收（掌控点 2）

- 功能是否符合预期
- 有没有引入新问题
- 在真实环境中验证过

### 7. Opus 合并

- 整理改动清单
- 更新项目文档
- 合并到主分支

## 你的三个掌控点

| 掌控点 | 时机 | 你做什么 |
|--------|------|---------|
| 需求对齐 | Spec 产出后 | 审 Spec，确保方向对 |
| 质量把控 | 审查通过后 | 验功能，确保实现对 |
| 最终决策 | 验收通过后 | 批合并，确保时机对 |

你不需要看每一行代码——那是 Opus + Coding Model 之间的事。你只需要在这三个节点做决策。

## 多项目并发

```
项目A                    项目B
  ├─ Spec A1               ├─ Spec B1
  ├─ Spec A2               ├─ Spec B2
  └─ Spec A3               └─ Spec B3

Opus 产出所有 Spec → 你统一排优先级 → Coding Model 按序执行
```

- 每个项目维护独立 Spec 队列
- 明确 Spec 间依赖（B2 依赖 B1）
- Coding Model 一次做一个 Spec

## 简化场景

| 任务类型 | 流程 |
|---------|------|
| 新功能（>2 文件） | 完整 7 步 |
| 小改动（1-2 文件） | 跳过 Step 3（你审Spec），其余不变 |
| Hotfix | Opus 快速出 Spec(5min) → 实现 → 审查 → 合并。跳过 Step 3，Step 6 不可省 |
| 单行修复 / typo | 不走流程，直接改 |

## 与 Vibecoding 7 原则的关系

| 原则 | 本流程中的位置 |
|------|--------------|
| 0. CLAUDE.md 先行 | Opus Step 2 的第一步 |
| 1. 先画蓝图 | Opus Step 2 产出 Spec |
| 2. 一次一件事 | Spec 粒度控制 |
| 3. 先写测试 | Coding Model Step 4 |
| 4. 代码审查 | Opus Step 5 |
| 5. MCP+Skills | 项目 CLAUDE.md 中配置 |
| 6. 上下文重置 | 每个 Spec 独立执行，天然隔离 |

## FAQ

**Q: Coding Model 上下文装不下整个项目怎么办？**
A: 不需要。Spec 已给出"涉及文件"和"参考代码"，只读这几份就够了。这就是 Spec 的核心价值。

**Q: Coding Model 实现和 Spec 有偏差？**
A: 偏差是更好方案 → 更新 Spec。偏差是自作主张 → REWORK。偏差是 Spec 写得不清楚 → Opus 改进。

**Q: 我能跳过 Opus 直接让 Coding Model 干吗？**
A: 简单任务可以。判断标准：涉及 >2 文件或需要架构判断 → 必须走 Spec。

**Q: 这套流程适合个人项目吗？**
A: 适合。个人项目的话，"你"同时是 PM 和 merge 负责人，规则更简单。
