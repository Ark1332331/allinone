# Claude 并行开发边界清单

> 用途：当你准备把一个并行任务交给 Claude 时，先看这份清单，避免上下文污染、文件冲突和责任模糊。

## 1. 什么时候适合交给 Claude

适合：

- 明确独立的实验模块
- 不阻塞当前主线功能
- 写入边界可以清楚列出来
- 即使失败也不会污染主线

不适合：

- 当前主线正在频繁改动的核心文件
- 需要先和用户反复对齐的方案设计
- 依赖大量隐含上下文的任务
- 一旦出错会拖垮主链路的改动

## 2. 交付前必须写清楚的四件事

每次交给 Claude 前，至少明确：

1. 任务目标
   这次到底要交付什么，不要写成泛泛“接入一下”。
2. 允许改哪些文件
   最好列到文件级别。
3. 不允许改哪些文件
   尤其是当前主线区域。
4. 验收标准
   交付完成后，什么算合格，什么算跑偏。

## 3. 推荐任务单模板

可以直接复用下面这段：

```text
先读：
1. docs/README.md
2. docs/context.md
3. docs/active/current-focus.md
4. docs/active/handoff.md
5. 这次任务相关 spec
6. AGENTS.md

任务目标：
[一句话写清楚这次要交付什么]

只允许改：
- [文件 A]
- [文件 B]

不要改：
- [文件 X]
- [文件 Y]

验收标准：
1. [标准 1]
2. [标准 2]
3. [标准 3]

如果你发现设计会明显改变用户工作流，先停下来，不要默默拍板。
```

## 4. 当前仓库的高风险区域

这些区域默认不要随便交给 Claude：

- `api/api.py`
- `api/prompts.py`
- `api/routes/pre_assessment.py`
- `api/services/pre_assessment.py`
- `api/models/pre_assessment.py`
- `src/app/pre-assessment/`
- `src/components/PreAssessmentCards.tsx`
- `src/types/pre-assessment.ts`
- `docs/specs/pre-assessment-plan.md`

原因不是“Claude 一定做不好”，而是：

- 这些地方和主线耦合高
- 当前真相还在收敛
- 改动后很难快速分清是谁引入的问题

## 5. 当前适合并行外包的区域

更适合交给 Claude 的类型：

- 实验性 provider
- 独立工具接入层
- 辅助脚本
- 非主线文档
- 归档整理

前提是：

- 命名上要带实验性标记
- 不碰主线功能入口
- 失败可回滚

## 6. Claude 交付后，Codex 应该怎么接

不要直接相信“已完成”。

Codex 接手时要检查：

1. 是否真的只改了允许改的文件
2. 是否新增了隐性耦合
3. 是否把实验能力伪装成正式能力
4. 是否把阻塞描述成了事实，但其实已经不是事实
5. 是否真的能导入、能跑、能说明白

## 7. 你的角色

你不需要替两个 agent 手工转述大量聊天内容。

你只需要做三件事：

1. 指定优先级
2. 选谁负责哪一块
3. 在 Codex 审核后决定是否接受

如果一个任务需要你反复做人肉转发，说明协作结构本身还没搭好。
