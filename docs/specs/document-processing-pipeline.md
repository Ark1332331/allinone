# 文档处理流水线与解析策略

> Status: draft
> Owner: Codex + User
> Date: 2026-05-13

## 1. 为什么要单独定义这一层

当前一个容易让人误判的地方是：

看见某些页面里展示出的 markdown 很整齐，就误以为“pdf -> md”本身天然就能做得很干净。

这通常不成立。

更真实的情况往往是：

`原始文档 -> 解析 -> 结构重建 -> 清洗 -> 章节切分 -> 阅读展示`

也就是说，用户最终看到的整齐 md，通常不是第一次转换结果，而是流水线产物。

如果我们不把这层说清楚，后面就会重复发生两类错误：

- 过度高估转换器输出质量
- 把 LLM 错当成底层解析器

## 2. 基本判断：md 不是唯一真相

后续系统不应该只保存一份 markdown，然后把所有能力都压在它上面。

至少应保留三层结果：

1. 原始解析层
   - 页码
   - 顺序
   - 块类型
   - 原始抽取文本
   - 可能的定位信息
2. 清洗后的可读层
   - 给用户阅读
   - 给选区提问
   - 给目录跳转
3. 结构层
   - 章 / 节 / 小节
   - 段落块 / 公式块 / 表格块
   - 阅读器树

结论：

`markdown` 是面向阅读和交互的表现层，不应该被当成唯一数据层。

## 3. 文档处理中最常见的四类问题

### 3.1 文本识别问题

- OCR 识别错字
- 扫描页质量差
- 数学符号丢失

### 3.2 阅读顺序问题

- 双栏错序
- 页眉页脚混进正文
- 脚注和正文打乱

### 3.3 结构识别问题

- 标题层级错
- 列表被打平
- 表格被压扁
- 图注和正文混在一起

### 3.4 跨块语义问题

- 跨页段落断裂
- 公式前后文脱节
- 一章被错误切开

LLM 适合处理其中一部分，但不能承担全部。

## 4. LLM 在这条流水线里的正确位置

LLM 更适合做：

- 断行合并
- 页眉页脚去噪
- 标题层级规范化
- 章节切分建议
- 列表和段落重排
- 给“不确定块”打标

LLM 不应被默认用来：

- 直接从脏 PDF 一次性重写整本书
- 无证据改公式
- 自由臆断表格结构
- 无约束重写原文逻辑

结论：

LLM 是“后处理整理器”，不是“底层解析器”。

## 5. 为什么当前不默认全量使用重型解析器

长书、高公式密度、复杂 PDF 确实更适合重型结构解析路线。

但当前阶段如果所有文件都强行走重型路线，会有几个问题：

- 本地依赖更重
- 启动和部署复杂度更高
- 小文件导入收益不成比例
- 一旦底层解析器卡住，整个导入层都会被拖慢
- 当前最核心的产品问题会被基础设施问题掩盖

所以当前更合理的策略不是“永远不用重型解析器”，而是：

`按材料类型和使用目标分流`

## 6. 当前建议的两条解析路线

### 6.1 轻量路线

适合：

- md / txt
- html
- docx
- pptx
- 较短、排版不复杂的 pdf
- 需要快速进入问答的零碎材料

流水线：

`导入 -> 轻量转换 -> 基础清洗 -> 初步结构 -> 阅读/问答`

特点：

- 快
- 易接入
- 易联调
- 对小材料性价比高

### 6.2 高保真路线

适合：

- 教材
- 习题书
- 长篇 PDF
- 公式多
- 多栏排版
- 需要稳定章节树和阅读器定位

流水线：

`导入 -> 结构解析 -> 块级中间层 -> LLM 清洗/重组 -> 章节树 -> 阅读器`

特点：

- 更适合电子书模式
- 更适合后续做进度、锚点定位、章节问答
- 代价也更高

## 7. 处理策略应由什么决定

不应只由文件扩展名决定。

至少应综合：

1. 材料长度
2. 版面复杂度
3. 公式/表格密度
4. 当前目标
5. 是否要进入持续阅读

例如：

- 一个 20 页的简单 PDF
  可能适合轻量路线
- 一本 900 页教材
  即使也是 PDF，也更适合高保真路线

## 8. 当前最小中间层建议

后端文档处理中，至少建议产出下面这些字段：

- `raw_blocks`
  原始块
- `clean_markdown`
  清洗后可读文本
- `structure_tree`
  章节树
- `quality_report`
  清洗/结构化质量说明
- `warnings`
  例如公式疑似损坏、表格失真、章节识别不稳

这样后续前端能做三件重要的事：

1. 正常阅读
2. 对低质量区域做提示
3. 必要时回退到原始块视图

## 9. 关于“整洁 md”的现实判断

这里直接给真实判断，不做安慰：

- 对 PDF 来说，100% 干净整洁，不现实
- 对很多材料来说，做出“足够清晰可读、可分层、可提问”的 md，是现实目标
- 真正该追求的不是“每一份 md 都完美”，而是：
  - 可定位
  - 可阅读
  - 可问答
  - 可迭代修复

## 10. 与后续阅读器的关系

文档处理流水线最终不是为了“导出一个好看的 md 文件”，而是为了支撑：

- 学科资料架
- 阅读器目录
- 章节定位
- 选区问答
- 历史恢复
- 掌握度和画像更新

所以流水线设计必须服务后续对象模型，而不是只服务转换结果。

## 11. 当前阶段最适合先落地什么

当前优先级建议：

1. 先把文档处理分成“轻量路线 / 高保真路线”
2. 先定义中间层 schema
3. 先让现有轻量路线输出质量说明和结构树雏形
4. 等阅读器对象模型稳定后，再决定高保真路线具体接哪种解析器

## 12. 当前建议结论

当前文档处理策略不应被简化成：

- “都转 md 就好了”
- “加一个 LLM 整理就好了”

更合理的结论是：

1. `md` 是表现层，不是唯一真相
2. LLM 负责后处理，不负责承担底层全部解析
3. 材料长度和复杂度不同，解析策略也应不同
4. 当前最重要的是让流水线服务对象模型和阅读器，而不是单独追求“转换器看起来很强”
