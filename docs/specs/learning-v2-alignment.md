# 学习助手 v2 对齐方案

> Status: active draft
> Owner: Codex + User
> Date: 2026-05-14

## 1. 为什么现在不适合继续直接实现

当前问题已经不再是“缺几个功能”或者“页面不好看”，而是产品抽象层级还没有彻底对齐。

如果继续直接实现，会继续重复这几类错误：

- 把材料类型、学习目的、交互方式混在一页里
- 把 `pre-assessment` 误当成所有学习场景的统一入口
- 把 `markdown` 误当成所有资料的天然主载体
- 把助手内部记忆直接泄露到用户前端，或者反过来完全不可校对
- 在还没说清“系统到底在帮什么忙”之前就继续堆页面和接口

因此当前阶段应该暂停“继续补功能”，先完成一次正式的 v2 对齐。

## 2. v2 的产品定义

`allinone` 当前更准确的定义不是“一个前置评估页”，而是：

- 一个按学科组织的学习助手
- 一个围绕资料持续阅读、提问、复习、串联的工作空间
- 一个会观察用户、更新判断、必要时引导和纠偏的长期助手

它不应该再主要表现为：

- 一个要求用户先写清自己卡点的专家表单
- 一个只会做“值不值得读”判断的单点工具
- 一个把所有学习都塞进同一条流程的入口页

## 3. 四条必须拆开的轴线

后续所有设计都必须显式区分这四条轴线，不能再混写。

### 3.1 材料类型

- 教材 / 电子书
- 习题书 / 题单
- PPT / 课件
- 讲义 / 笔记
- 往年卷
- 考纲 / 重点提纲 / 老师要求

### 3.2 学习目的

- 复习提分
- 首次理解
- 长期自学 / 兴趣探索
- 刷题纠错

### 3.3 交互层

- 导入资料
- 阅读资料
- 划线提问
- 多轮追问
- 初步判断
- 复习规划
- 进度恢复

### 3.4 证据来源

- 材料正文
- 图示 / 版式 / 公式位置
- 往年卷
- 老师重点
- 考核标准
- 用户过往提问与停留行为

## 4. 学习模式重新定义

当前至少需要三种模式，但模式是“默认工作重心”，不是底层交互能力边界。

### 4.1 复习提分模式

核心目标：

- 不是完整掌握全部内容
- 而是在有限时间内提高得分产出比

它必须能吃进的资料：

- 往年卷
- 老师画重点
- 平时练习题
- 考纲 / 考核标准
- 课件 / 教材相关章节

它优先回答的问题应该是：

- 这门课当前最该抓什么
- 哪些内容最可能决定分数
- 这份资料在当前备考里扮演什么角色
- 哪些章节先过，哪些章节先不深究

### 4.2 理解探索模式

核心目标：

- 帮用户建立结构化理解
- 降低进入陌生材料的门槛

它优先回答的问题应该是：

- 这份材料到底在讲什么
- 先看什么能最快抓住骨架
- 哪些前置概念缺失会导致后面一直卡
- 现在读这份材料值不值得

### 4.3 电子书阅读模式

核心目标：

- 让“阅读过程本身”变成主要工作区

核心能力：

- 目录导航
- 页内定位
- 划线提问
- 问答历史恢复
- 多轮追问
- 下次从锚点继续

补充说明：

- 它不是唯一能划线提问的模式
- 但它最强依赖“阅读器 + 锚点 + 历史恢复”

## 5. 一个关键纠偏：划线提问是通用交互层，不是模式专属

后续实现必须把 `划线提问` 从模式层剥离。

只要界面中存在可引用文本，就应该支持：

- 选区
- 提问
- 继续追问
- 关闭后恢复
- 将问答锚定到原位置

适用对象包括：

- 原始材料正文
- 清洗后的 markdown
- 用户笔记
- 系统摘要
- LLM 回复

## 6. 文档处理的核心结论：md 只能是辅助层，不是唯一主层

### 6.1 哪些内容适合主要走 md

- 纯文本讲义
- 短笔记
- 结构较简单的资料
- 公式和版式要求不高的文档

这类资料可以优先走：

`导入 -> 转 markdown -> 清洗 -> 阅读/提问`

### 6.2 哪些内容不能只靠 md

- 数学教材
- 带大量公式、图示、例题布局的 PDF
- PPT / PPTX / 导出 PDF 课件
- 强依赖插图、箭头、版式强调的材料

即使能转出文字，也会丢这些真正重要的信息：

- 图片
- 页面布局
- 公式相对位置
- 图和正文的对应关系
- PPT 中的视觉强调

### 6.3 v2 应采用双通道

#### 源文件阅读层

- PDF / PPT / PPTX 原始页视图
- 页级定位
- 截图或框选提问

#### 结构化理解层

- markdown 文本
- OCR 结果
- 章节树
- 段落块
- 公式文本
- 术语 / 题目块

作用：

- 检索
- 摘要
- 前置判断
- 复习规划
- 跨资料串联

## 7. 截图提问为什么已经进入主需求

对于数学、PPT、图示型资料，截图提问不是锦上添花，而是主需求。

因为很多真实问题并不来自纯文本本身，而来自：

- 图和公式的对应关系
- 某一步推导在版面中的位置
- 某张 PPT 上被高亮的元素
- 例题图形本身

所以后续阅读器设计必须考虑：

- 页面视图
- 区域截图
- 截图与文本块双向引用
- 视觉区块的问答锚定

## 8. 记忆机制改成双层：内部层 + 可见校对层

用户已经明确指出两个问题：

- 不应该把类似“内部提醒自己”的内容展示在前端主界面
- 但也不能完全看不见更新了什么，否则无法监督助手是否真的在观察与学习

因此 v2 的记忆机制建议如下。

### 8.1 内部层

- 位置：`.codex/memory/`
- 用途：稳定画像、当前协作状态、项目方向、助手契约
- 原则：文件少而稳定，不按每次对话无限膨胀

### 8.2 可见校对层

- 本地中文文档
- 不放到学习产品主页面
- 用于告诉用户：
  - 这次观察到了什么
  - 更新了什么判断
  - 这会如何影响后续行为
  - 哪些内容仍待确认

## 9. 以后如何避免“我以为你懂了，你却已经开始实现另一条路”

后续涉及核心能力时，默认采用这个顺序：

1. 先明确我正在实现哪条路
2. 明确这条路服务什么场景，不服务什么场景
3. 明确这条路当前还需要你提供什么
4. 你确认后，再进入实现

尤其当问题涉及下面这些层级时，不能再直接跳实现：

- 产品主流程
- 材料处理策略
- 记忆机制
- 阅读器交互
- 学习模式拆分

## 10. 当前阶段用户需要最少提供什么

### 10.1 学科样本

- 一门专业课复习资料组合样本
- 一份数学/深度学习类阅读样本

### 10.2 复习提分模式的证据样本

至少一种即可：

- 往年卷
- 老师重点
- 考纲
- 平时题单

### 10.3 视觉材料样本

至少一种即可：

- 数学 PDF
- PPTX
- 带图的课件 PDF

用户不需要在这一阶段先把“自己卡在哪”写成表单。

## 11. v2 这一阶段不该继续做什么

- 不继续围绕旧 `/pre-assessment` 页面补表单
- 不先追求页面美化来掩盖抽象混乱
- 不继续把所有学习都压进一条入口链路
- 不继续把 md 当唯一工作对象
- 不把内部记忆展示混进主用户界面

## 12. 下一阶段的明确产物

在真正恢复实现前，下一阶段应该先完成这 5 个产物：

1. 学习对象模型定稿
2. 文档处理双通道方案定稿
3. 学科首页 -> 资料架 -> 模式路由的页面骨架图
4. 阅读器最小闭环定义
5. 记忆变更台账机制定义

## 13. 当前建议结论

当前最合理的主线不是“继续补功能”，而是：

`先完成 v2 对齐 -> 再选择一个模式做最小闭环`

并且当前最值得优先验证的闭环不是旧的 `pre-assessment`，而是：

- 学科页
- 资料架
- 阅读器入口
- 视觉资料 / 文本资料双通道
- 通用划线提问

`pre-assessment` 应保留，但它应该退回为某些模式中的一个能力节点，而不是继续占据产品主入口。

## 14. 本轮新增对齐结果

这一轮对齐补上的不是更多功能，而是把主路径、资料对象、背景参与方式和阅读页交互真正收紧。

### 14.1 主路径正式收口

当前默认主路径应为：

`进入学科 -> 查看资料库 -> 点击资料卡 -> 直接进入正文阅读 -> 划线提问 / 连续追问`

补充约束：

- 不再经过“资料详情中间页”
- `pre-assessment` 不再承担主入口职责
- 所有资料默认都先走同一条阅读主路径，包括教材、课件、往年卷、考纲、重点提纲、复习资料

### 14.2 `pre-assessment` 的新位置

`pre-assessment` 现在应被理解为阅读页中的一个可选能力，而不是独立入口页。

阅读页中它的形态应为：

- 顶部按钮
- 点击后弹出短判断层
- 即使用户不点开，相关判断也可作为右侧追问区的隐式背景

面向用户的可见字段应收敛为：

- 现在适合怎么用
- 先看哪里
- 先跳过哪里
- 当前最大阻塞
- 高复用骨架

可折叠部分再展示：

- 为什么这样判断
- 判断主要来自材料本身、当前目标，还是用户长期画像
- 把握高低与待验证项

### 14.3 资料对象模型的两层拆分

后续对象建模不能再把“文件长什么样”和“这份资料在学习中扮演什么角色”混在一起。

必须拆成两层：

- 文件类型：决定解析与阅读技术路径
- 学习角色：决定产品行为与默认辅助方式

当前已确认的学习角色至少包括：

- 教材/主干内容
- 新课课件
- 复习课件
- 重点提纲/老师划重点
- 考纲/考核要求
- 往年卷
- 习题/作业/题册
- 讲评资料/答案解析

其中 `复习课件` 不应再被等同于普通课件。

### 14.4 导入后的默认分类策略

不追求让用户在导入时手动把资料分完。

当前确认的导入策略是：

- 系统先猜 `主定位`
- 系统可附带少量 `辅助定位`
- 用户只做轻确认，而不是手动填写完整分类

补充约束：

- 每份资料有且仅有一个 `主定位`
- 每份资料可以带少量 `辅助定位`

### 14.5 当前目标与背景资料

学科内需要允许用户维护“当前这轮学习/复习到底以什么为准”。

当前确认的设计是：

- 用户手动创建和切换 `当前目标`
- 目标名称不预设为期中/期末/测验，可由用户自由命名
- 资料库中的资料可以一键加入/移出 `当前目标背景`
- 这个开关默认放在资料卡上，正文顶部只做同步显示

并且，以下资料不只是“可阅读资料”，还应成为判断背景：

- 往年卷
- 考纲
- 老师划重点
- 复习资料

它们会影响助手对其他资料的判断，而不是只作为孤立文本存在。

### 14.6 背景资料的作用分类

加入 `当前目标背景` 时，不再让系统偷偷乱猜其判断职责。

当前确认的规则是：

- 用户手动选择背景作用
- 每份背景资料只能选择一个作用

当前作用类型固定为三类：

- 标准类：决定考什么、重要度如何
- 证据类：决定哪些题型/考法真实出现过
- 讲解类：决定如何理解、如何压缩

### 14.7 资料库默认展示规则

资料库首页当前不做“系统自作主张的智能优先级排序”。

现阶段默认规则应为：

- 全部混排
- 不默认按当前目标强行重排

每张资料卡当前默认只展示：

- 资料名
- 主定位
- 辅助定位
- 是否加入当前目标背景
- 最近阅读时间

### 14.8 阅读页的默认上下文栈

阅读页右侧追问区不应只围绕当前文字片段工作。

当前确认的默认上下文栈是：

- 硬上下文：当前选中的正文片段
- 强背景：当前目标下已激活的背景资料
- 软背景：用户长期画像

这三层一起构成阅读页和普通网页 LLM 的本质区别。

### 14.9 判断必须可查账

助手在阅读页中给出的判断不能是黑箱。

当前确认的审计要求是：

- 默认折叠显示
- 用户可随时展开查看依据

展开后至少应能看到：

- 参考了哪些背景资料
- 结论更偏材料本身、当前目标，还是长期画像
- 哪些判断把握高
- 哪些判断只是推测
- 如果结论不对，应优先纠正哪一层

### 14.10 多处选区提问，但暂不跨资料

阅读页的提问模型不能再被限制为“单段文字即时提问”。

当前确认的范围是：

- 支持在同一份资料中选中不止一处正文
- 多个片段可以临时组成一个提问上下文包
- 再结合强背景与软背景进行统一提问

但本阶段明确不做：

- 跨资料选区拼接提问
- 跨资料证据拖拽式组装

也就是说，当前实现范围是：

`多处选区 -> 仅限当前资料内部 -> 统一提问`
