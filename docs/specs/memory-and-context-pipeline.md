# Memory And Context Pipeline

## 目标

把当前“Codex 读文档再协作”的经验迁移到 allinone 的模型 API 调用里。

核心目标不是把所有文档塞进 prompt，而是让模型在合适时刻拿到合适层级的上下文：

1. 稳定画像：长期有效、短、经过用户确认或多次验证。
2. 学习痕迹：原始证据，可追溯，但不直接当成人格结论。
3. 当前任务状态：当前学科、资料、目标、选区、问答线程。
4. 协作规则：模型必须如何对待用户、如何避免猜测、何时暂停。

## 非目标

- 不做不可见、不可校对的自动画像写回。
- 不把单次纠偏直接升级成稳定用户画像。
- 不让前端主流程暴露一堆“记忆管理”复杂度。
- 不把所有历史对话全文长期塞进每次模型调用。

## 上下文分层

### 1. 协作规则层

来源：

- `AGENTS.md`
- 未来产品内置 system/developer prompt

用途：

- 约束模型行为，而不是描述用户。
- 例如：
  - 先搞清用户真正卡点，不脑补。
  - 连续修错两次必须先做根因链路。
  - 重要观察先进入学习痕迹，不直接写入稳定画像。

调用方式：

- 每次关键 API 调用都应加载压缩后的协作规则。
- 规则必须短，不应包含历史流水。

### 2. 稳定画像层

来源：

- 未来的用户画像文件或数据库表。
- 只接受用户确认或多次学习痕迹支持的结论。

用途：

- 帮助模型选择解释深度、学习路径和建议方式。
- 例如：
  - 用户偏好先看结构和边界，再深入细节。
  - 用户希望助手能质疑、纠正，而不是被动顺从。

调用方式：

- 默认加载精简版。
- 对具体任务可按领域加载相关片段。

### 3. 学习痕迹层

来源：

- `docs/active/learning-traces.md`
- 未来可迁移为可检索记录表。

用途：

- 保留原始证据。
- 为模型判断“这次是否重复出现某类能力缺口”提供依据。
- 为用户提供可见校对入口。

调用方式：

- 不默认全量加载。
- 根据当前任务检索相关记录，例如“前端交互调试”“代码理解”“数学阅读”。
- 命中后只摘要证据和建议，不当作永久人格标签。

### 4. 当前任务状态层

来源：

- 当前页面路由
- 当前学科/课程
- 当前资料对象
- 当前学习目标
- 当前选区、截图、锚点线程
- 用户手动选定的背景资料

用途：

- 回答当前问题必须依赖的硬上下文。
- 这是优先级最高的业务上下文。

调用方式：

- 每次学习问答都必须带：
  - 当前材料 id/title/type
  - 当前选区或截图描述
  - 当前问答线程
  - 当前目标背景
- 大材料不全量塞入 prompt，优先传选区、附近上下文、结构摘要和可检索片段。

## 调用组装顺序

建议 API 调用时按以下顺序组装上下文：

```text
1. 协作规则摘要
2. 稳定画像摘要
3. 与当前任务相关的学习痕迹摘要
4. 当前学科/资料/目标状态
5. 当前选区、截图或用户问题
6. 明确输出要求
```

优先级：

```text
当前硬上下文 > 用户本轮明确要求 > 协作规则 > 稳定画像 > 学习痕迹 > 历史归档
```

如果稳定画像或学习痕迹与用户本轮表达冲突，必须相信用户本轮表达，并可生成一条“画像可能需要修正”的候选记录。

## 画像更新流程

模型回答后可以生成候选观察，但不能静默写入稳定画像。

推荐流程：

```text
对话/学习事件
-> 生成学习痕迹候选
-> 用户可见确认/修改/拒绝
-> 进入学习痕迹
-> 多次重复或用户明确认可
-> 提炼为稳定画像候选
-> 用户确认
-> 写入稳定画像
```

候选学习痕迹格式沿用：

```text
日期：
触发事件：
用户原话/现象：
观察到的能力或缺口：
证据：
下一次协作建议：
对产品设计的启示：
是否进入稳定画像：否 / 待验证 / 是
```

## 面向代码理解功能的迁移

代码理解功能不应只解释文件，而应生成“交互链路图”：

```text
用户动作
-> 涉及组件
-> 事件监听器
-> 状态写入位置
-> 渲染位置
-> 可能断点
-> 验证方式
```

当用户描述 UI/代码现象时，模型应优先问或推断：

- 场景：在哪个页面、哪个框、哪个材料形态？
- 动作：用户做了什么？
- 预期：用户希望哪里发生什么？
- 实际：现在哪里发生了什么？
- 路径：实际代码可能走了哪条组件/事件/状态链？

这正是从 vibecoding 过渡到“能和代码对话”的中间层。

## 工程落点

短期：

- 保持 `docs/active/learning-traces.md` 作为人工可见原型。
- 保持 `docs/active/handoff.md` 精简，只放当前合同。
- 把历史流水归档到 `docs/archive/handoffs/`。

中期：

- 增加用户可见的“学习痕迹候选”确认 UI。
- 为学习痕迹加标签，如 `frontend-events`、`debugging`、`math-reading`、`code-understanding`。
- API 调用前按标签检索相关痕迹。

长期：

- 建立稳定画像表、学习痕迹表、当前任务状态表。
- 每次模型调用由 context builder 统一组装，不让各功能随意拼 prompt。
- 所有画像更新都可见、可撤销、可追溯到原始学习痕迹。
