# 学习模式分流对齐

> Status: draft
> Owner: Codex + User
> Date: 2026-05-13

## 1. 为什么必须分流

当前产品把不同学习目的压进了同一条“导入 -> 前置判断”链路，这会导致判断提示词、界面形态、输入材料、输出价值全部错位。

至少要分成三类主模式：

1. 复习提分模式
   面向专业课、考试复习、分数提升。
2. 理解探索模式
   面向深度学习、现代 AI、概念性阅读，不以短期分数为唯一目标。
3. 电子书交互模式
   面向教材、习题书、长篇 PDF，强调正文阅读、划线提问、锚点问答、多轮追问。

结论：
后续不能再把 `pre-assessment` 默认当成所有学习行为的统一入口，只能把它视为某些模式中的一个能力节点。

## 2. 三类模式的核心差异

### 2.1 复习提分模式

核心目标：
- 帮用户提高考试成绩，而不是只讲“值不值得读”

典型输入：
- 课件
- 重点范围
- 往年卷
- 老师强调点
- 习题/作业

核心输出：
- 高频考点
- 本轮复习优先级
- 可能出题的知识块
- 缺失的关键前置
- 针对分数目标的复习路径

这一模式下更重要的提示词不是：
- “值不值得读”
- “先补哪些概念”

而是：
- “哪些知识最可能决定分数”
- “哪些题型/考点最值得优先投入”
- “这份材料该如何服务当前复习目标”

### 2.2 理解探索模式

核心目标：
- 帮用户建立结构化理解，降低进入陌生材料的门槛

典型输入：
- 技术书
- 研究型材料
- 陌生但重要的课程资料

核心输出：
- 这份材料到底在讲什么
- 先读什么，后读什么
- 哪些概念是前置骨架
- 哪些细节可以暂缓
- 当前投入是否划算

这一模式下，当前的：
- “补概念”
- “值不值得读”
- “先骨架后深入”

这些能力依然是核心。

### 2.3 电子书交互模式

核心目标：
- 导入整本书后，把阅读过程本身变成可交互对象

典型输入：
- 微积分教材
- 习题书
- 专业教材
- 长篇 PDF / lecture notes

核心输出与交互：
- 电子书式阅读界面
- 按目录浏览正文
- 任意位置划线
- 对高亮位置提问
- 多轮追问
- 关闭弹窗后仍保留该锚点问答
- 重新点击对应句子时恢复历史问答

这一模式下，最关键的不是“前置判断页”，而是：
- 书架
- 阅读器
- 锚点问答侧栏
- 问答持久化

## 3. 建议的信息架构

建议从“单一学习入口页”改为“学科 -> 资料架 -> 模式路由”。

### 3.1 顶层

- 学科主页
  例如：高数 / 概率论 / 深度学习 / 专业课 A

### 3.2 学科内资料架

每个学科下允许导入：
- 书籍
- 习题
- 课件
- 往年卷
- 重点提纲

### 3.3 进入资料后的模式路由

系统先根据材料类型与用户目标推断默认模式：

- 课件 + 往年卷 + 复习目标
  默认走“复习提分模式”
- 技术书 / 新课 / 概念材料
  默认走“理解探索模式”
- 整本教材 / 习题书 / 长篇 PDF
  默认走“电子书交互模式”

用户仍可切换，但默认路径必须更聪明。

## 4. 当前阶段最应该做什么

不是继续往 `pre-assessment` 页里堆更多字段，而是先明确下一阶段主线：

1. 完成“模式分流”产品定义
2. 确定 v2 首先做哪一个模式
3. 明确三种模式的最小可用闭环

建议优先级：

1. 先定模式分流
2. 再做电子书交互模式的最小阅读器
3. 最后把复习提分模式做成更强的材料组合分析

原因：
- 电子书交互模式最贴近用户现在给出的理想使用图景
- 它也最能检验“助手是否真的跟着用户阅读”
- 它会自然暴露个性化、记忆、锚点问答等能力的真需求

## 5. 如何验证“助手真的个性化且分析到位”

不能只靠主观感觉，要建立可验证机制。

### 5.1 交互层验证

每次关键分析至少要能回答三件事：

1. 我为什么这样判断
2. 我依据的是材料哪里 + 你的哪些长期特征
3. 这次建议和上次相比，有没有因为你这个人而变化

### 5.2 记忆层验证

每次多轮使用后，应能看到：

- 新增了哪些用户画像
- 哪些旧判断被修正
- 哪些建议明显开始“更像是给你，而不是给任何人”

### 5.3 效果层验证

至少建立两类验收：

- 复习提分模式
  看建议是否帮助缩小复习范围、提高做题命中率、减少无效投入
- 理解探索 / 阅读器模式
  看建议是否帮助更快抓骨架、减少卡顿、形成持续问答链

## 6. 对当前阶段的批评

当前版本的问题不是“功能太少”，而是：

- 过早把不同学习目标揉成了一条流程
- 默认假设用户应该先明确自己的卡点
- 还没建立“阅读过程本身就是主界面”的思路

这不是细节偏差，而是产品抽象层级偏差。

## 7. 下一步建议

下一阶段应先做一次正式对齐，主题不是“再加什么字段”，而是：

1. 三类模式是否成立
2. v2 主推哪个模式
3. 学科主页 / 资料架 / 阅读器 / 复习模式之间怎么接
4. 哪一部分由 Claude 执行，哪一部分由 Codex 负责架构审查与对齐
