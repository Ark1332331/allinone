# 学习入口工作流（Learning Entry Workflow）

> Status: active draft
> 这个文档描述当前更高优先级的产品主路径。
> `pre-assessment` 仍保留，但它不再被视为完整入口，而是学习入口中的一个判断节点。

## 1. 为什么要重写主路径

当前 `pre-assessment` v1 虽然已经从“难度分类器”收窄成“学习决策助手”，但它仍然保留了过强的“专家工具”气质：

- 需要用户自己准备完整材料
- 需要用户先写目标提示
- 需要用户先说清当前卡点
- 需要用户先组织输入，系统才开始工作

这和当前真实目标冲突。

当前要做的不是“让用户更好地填写表单”，而是“让系统更早地替用户承担整理、暴露结构、降低提问门槛的工作”。

## 2. 当前真正的主目标

主目标不是单独做一个前置评估页面。

主目标是建立一条低摩擦学习入口：

1. 用户把材料丢进来
2. 系统自动转成可读 markdown
3. 系统先暴露材料结构和初步判断
4. 用户在阅读过程中再逐步暴露真实卡点
5. 助手根据阅读行为、提问方式和卡点表现，持续观察并更新对用户的理解

也就是说，真正的入口应该是：

`导入 -> 转换 -> 初判 -> 阅读 -> 划线/提问 -> 暴露卡点 -> 个性化引导`

而不是：

`填写表单 -> 提交 -> 看结论`

## 3. 产品定位修正

`allinone` 当前最该对齐的定位是：

- 一个帮助用户进入学习状态的助手
- 不是一个要求用户先自我诊断完再来使用的工具

所以系统的责任前移：

- 不只回答问题
- 不只总结材料
- 不只输出评估
- 而是先帮用户“把学习入口打开”

## 4. v1 主路径

### 4.1 推荐的 v1 路径

1. 导入材料
   - 支持 pdf / ppt / docx / xlsx / md / txt
2. 自动转 markdown
   - 优先复用成熟轮子，例如 `markitdown` / `makeitmd` 同类方案
3. 展示材料骨架
   - 标题、章节、关键结构、适合先看的部分
4. 给出初步学习判断
   - 值不值得现在投入
   - 如果不适合，最阻塞的 1-2 个缺口是什么
   - 如果适合，先抓什么
5. 进入阅读态
   - 用户可以直接读
   - 用户可以选中一段再问
   - 用户可以在“这里看不懂/这里想深挖/这里像重点”三类轻量入口中选择
6. 逐步形成更真实的个性化引导
   - 当前卡点从阅读交互里长出来
   - 目标也允许后置显化，而不是前置强填

### 4.2 `pre-assessment` 在新路径中的位置

`pre-assessment` 不删除，但降级为：

- 导入后自动触发的“初判节点”
- 或者阅读前的“快速判断卡片”

它不应再独立承担完整主入口的角色。

## 5. 为什么不能继续把“当前卡点”做成前置输入

因为这隐含了一个错误假设：

“学生在还没开始读之前，就能准确说出自己卡在哪里。”

这通常不成立。

更常见的真实情况是：

- 只感觉难
- 不知道难在哪
- 知道某段看不懂，但说不出概念名
- 误把表面问题当成真实阻塞点

所以“当前卡点”更应该来自：

- 选区行为
- 追问行为
- 重复停留位置
- 用户对系统结论的反驳

而不是一个前置文本框。

## 6. 为什么“目标提示”不能再是刚需

“目标提示”不是没用，而是不能成为系统工作的前提。

当前更合理的处理是：

- 允许为空
- 如果为空，系统先按通用学习入口工作
- 在阅读过程中再逐步追问更具体的意图

例如只在必要时追问：

- 你是想应付作业、备考、入门，还是准备深入理解？
- 你现在是想先看全局，还是先解决这段？

这样比让用户一开始写一段抽象目标文本更符合真实使用场景。

## 7. 助手职责修正

这个项目当前不只是在做页面和接口，也是在验证“真正的学习助手应该怎么工作”。

因此助手本身必须承担三层职责：

1. 工具职责
   - 完成导入、转换、评估、阅读辅助
2. 观察职责
   - 从用户的输入、提问、犹豫、误判里观察画像变化
3. 引导职责
   - 当用户把问题描述错、把方向抓偏、或过早想做复杂方案时，明确指出并纠偏

如果系统只做第 1 层，它仍然只是工具，不是当前要构建的助手。

## 8. 对现阶段实现的直接约束

### 8.1 现在不该优先做什么

- 不优先继续堆表单字段
- 不优先继续打磨旧 `/pre-assessment` 页面细节
- 不优先先做“好看 UI”
- 不优先扩成 repo / code 统一入口

### 8.2 现在该优先做什么

- 明确导入型主路径
- 明确文件转 markdown 的接入方式
- 明确“初判”和“阅读”之间的关系
- 明确轻量提问入口，而不是依赖用户自写长文本

## 9. 建议的前端形态

首页第一条主路径应更接近：

- 导入一个文件
- 看系统先帮我整理出了什么
- 从整理结果进入阅读和提问

而不是：

- 先打开一个评估表单
- 再自己粘贴一大段内容

### 9.1 第一屏应该出现什么

- 一个清晰的导入入口
- 一句解释系统会先帮你做什么
- 一个低压力的开始方式

例如：

- 丢一个 PDF/PPT 进来，我先帮你拆出结构、判断值不值得现在读

### 9.2 阅读中的轻量交互入口

优先考虑这类入口，而不是长文本表单：

- `这里看不懂`
- `这里像重点`
- `这里先不用深究吗`
- `这段和我之前学过的什么相关`

## 10. 与 Claude / Codex 协作的意义

后续无论由谁实现切片，都要遵守这个优先级：

1. 先保证产品路径对
2. 再决定 UI 怎么美化
3. 再决定哪些子功能拆给其他 agent

否则很容易出现“实现越来越多，但离真实使用越来越远”的假进展。

## 11. 当前建议结论

当前应把“学习入口工作流”视为主 spec，把原有 `pre-assessment` 视为其子模块。

下一阶段优先问题应改为：

1. 文件导入和自动转 markdown 怎么接进来？
2. 初判结果如何嵌进阅读入口，而不是孤立成一页？
3. 阅读中的轻量交互如何取代“用户先写清卡点”的旧输入方式？

在这三个问题没有对齐前，不建议继续围绕旧 `/pre-assessment` 页面做大量实现或美化。
