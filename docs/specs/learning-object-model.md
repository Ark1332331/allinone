# 学习对象模型与交互层设计

> Status: draft
> Owner: Codex + User
> Date: 2026-05-13

## 1. 为什么现在必须先定对象模型

当前如果继续只讨论“页面长什么样”或“先做哪种模式”，很容易再次把不同层级的问题混在一起：

- 学习模式是什么
- 划线提问属于谁
- 章节树怎么长
- 画像和掌握度如何更新
- 下次进入时如何串联历史

这说明我们现在缺的已经不是一个新功能，而是“系统到底在操作哪些对象、对象之间怎么关联”。

如果对象模型不先定清楚，后面会反复出现这些错误：

- 把划线提问错误地绑定到“电子书模式”
- 把 `pre-assessment` 错误地当成所有学习行为的起点
- 把用户画像更新做成一次性评语，而不是持续证据积累
- 把“当前阅读态”和“历史学习链”割裂开

## 2. 两层拆分：模式层 vs 交互层

这里先做一个关键纠偏：

### 2.1 模式层

模式层负责回答：

- 当前用户这次是在复习提分、理解探索，还是长文阅读
- 系统默认应该给哪类建议
- 界面的主组织方式应该偏向哪种任务

当前至少有三种模式：

1. 复习提分模式
2. 理解探索模式
3. 长文阅读 / 电子书模式

### 2.2 交互层

交互层负责回答：

- 用户如何在任何文本上提问
- 问答如何锚定到某一段
- 关闭后如何恢复
- 多轮追问如何延续
- 哪些行为进入画像和掌握度更新

结论：

`划线提问` 不属于电子书模式，它属于底层通用交互能力。

只要界面中有可引用文本，都应该支持：

- 划线
- 选区提问
- 多轮追问
- 问答恢复
- 问答证据沉淀

适用对象包括：

- 原始材料正文
- 系统清洗后的 markdown
- 用户自己的笔记
- 系统生成的摘要
- LLM 的历史回复

## 3. 当前最小对象集合

后续系统至少要明确以下对象。

### 3.1 Subject（学科）

作用：

- 作为长期学习上下文容器
- 连接材料、问答、掌握度、目标

示例：

- 高等数学
- 概率论
- 深度学习
- 人工智能导论

它不只是文件夹，而是用户长期学习链的顶层单位。

### 3.2 Material（材料）

作用：

- 表示一次导入或一个持续阅读对象

类型示例：

- 书籍
- 习题书
- 课件
- 讲义
- 往年卷
- 重点提纲
- 用户笔记
- 系统生成的整理稿

关键区分：

- `短材料`
  如 PPT、小讲义、几页 PDF、重点提纲
- `长材料`
  如教材、习题书、长篇 lecture notes

### 3.3 Document Structure（文档结构）

作用：

- 保存章节树和块级结构
- 让系统知道如何分层显示，而不是只拿一整坨 md

至少包含：

- 卷 / 章 / 节 / 小节
- 段落块
- 列表块
- 公式块
- 表格块
- 图注块

它是“阅读器目录”和“分段引用”的来源。

### 3.4 Text Block（文本块）

作用：

- 是交互层最基础的可引用对象

每个块至少要有：

- 内容
- 顺序位置
- 来源材料
- 所属章节
- 原始页码或定位信息
- 清洗质量状态

只要一个对象能显示为文本块，它就应该支持选区和挂载问答。

### 3.5 Anchor（锚点）

作用：

- 把问答线程绑定到某段具体文本或某个范围

锚点不应只服务原文，也可绑定到：

- 系统摘要中的一句话
- 用户笔记中的一段
- LLM 回复中的一段

锚点是“再次点击恢复历史问答”的基础。

### 3.6 Thread（问答线程）

作用：

- 表示围绕某个锚点展开的一组多轮问答

至少有两种线程：

1. 锚点线程
   对某个具体位置提问
2. 全局线程
   对当前材料或当前学科提问

线程要保留：

- 问答历史
- 所属锚点
- 所属材料
- 所属学科
- 生成时的模式上下文

### 3.7 Learning Mode Context（模式上下文）

作用：

- 记录当前这次交互以哪种学习目标为主

例如同一本书：

- 为考试复习时，系统更关心高频考点
- 为理解探索时，系统更关心骨架和前置

所以“模式”不是材料固有属性，而更像：

- 当前材料
- 当前目标
- 当前阶段

共同决定的上下文。

### 3.8 Mastery Evidence（掌握度证据）

作用：

- 不直接存“你会了/不会”这种粗暴结论
- 先存能支持判断的证据

证据来源示例：

- 哪类概念反复追问
- 哪些章节停留很久
- 哪些题型经常误判重点
- 哪些建议被采纳后不再重复卡住

先存证据，再归纳“掌握度变化”。

### 3.9 Profile Signal（画像信号）

作用：

- 记录用户更稳定的学习偏好与盲区

例如：

- 更喜欢先骨架后细节
- 不愿先手工组织输入
- 对流程摩擦很敏感
- 容易把产品结构问题误当成单个功能缺失

它与掌握度不同：

- `Profile Signal` 更像长期风格和偏好
- `Mastery Evidence` 更像具体学科里的掌握证据

## 4. 为什么 PPT 和整本 PDF 不能走同一套处理方式

它们都可以进入同一个对象体系，但默认工作流不能完全一样。

### 4.1 短材料

如：

- PPT
- 几页讲义
- 一套往年卷

重点是：

- 快速抽主线
- 快速判断重点
- 快速进入问答

它们更像“分析对象”。

### 4.2 长材料

如：

- 教材
- 习题书
- 长 PDF

重点是：

- 结构分层
- 章节浏览
- 阅读进度
- 锚点问答恢复
- 历史串联

它们更像“持续阅读对象”。

结论：

`pre-assessment` 在短材料里更像“快速起步判断”，在长材料里更像“进入阅读器前的结构初始化”。

## 5. 章节树如何服务后续交互

“自动分层分章节小章节”不是一个装饰功能，而是后面很多能力的前提。

它至少服务：

- 左侧目录导航
- 当前阅读定位
- 章节级总结
- 章节级掌握度
- 问答线程归档
- 下次继续时的恢复点

所以章节树不能只是前端临时渲染结果，而应该成为文档结构对象的一部分。

## 6. 如何更新画像和掌握度

这里要避免两个极端：

### 6.1 极端一：什么都不记

结果：

- 每次都像第一次见用户
- 无法形成“越来越像为你服务”的感觉

### 6.2 极端二：每次都直接写死结论

结果：

- 误判会长期污染系统
- 用户一旦阶段变化，旧结论会拖后腿

更合理的做法是：

### 6.3 两阶段更新

第一阶段：记录证据

- 你问了什么
- 你在哪停住
- 你反驳了什么建议
- 你更愿意接受哪种帮助

第二阶段：定期归纳

- 形成稳定画像更新建议
- 形成掌握度变化判断
- 标明证据来源和置信度

## 7. 下次如何串联

后续“继续学习”至少有三种串联轴：

1. 同一学科串联
   例如高数下的多份课件、教材、题集互相连接
2. 同一材料串联
   例如上次读到第 5 章，这次继续
3. 同一知识点串联
   例如“贝叶斯”“卷积”“拉格朗日乘子”跨材料复现

这也是为什么对象模型必须先有：

如果没有 `Subject / Material / Anchor / Thread / Evidence` 这些对象，后续串联只能停留在聊天记录层，无法稳定工作。

## 8. 当前阶段的直接约束

在对象模型没有写清楚前，不建议直接重投入以下实现：

- 完整阅读器
- 自动掌握度面板
- 大规模画像自动写回
- 模式级复杂路由 UI

当前更适合先做：

1. 明确对象模型
2. 明确文档处理流水线
3. 再决定最小阅读器应该最先承载哪些对象

## 9. 当前建议结论

当前最重要的结论有三个：

1. `划线提问` 是通用交互层，不属于某一种模式
2. `短材料` 和 `长材料` 进入同一体系，但默认工作流不同
3. 后续实现应围绕对象模型推进，而不是继续围绕单页功能堆叠
